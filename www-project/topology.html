Topology

Here are a few questions about the underlying connectivity of a hypertext web.

Are links two- or multi-ended?

The term "link" normally indicates with two ends. Variations of this are links with multiple sources and/or multiple destinations, and constructs which relate more than two anchors. The latter map onto logic description systems, predicate calculus, etc. See the "Aquanet" system from Xerox PARC - paper at HT91. This is a natural step from hypertext whose links are typed with semantic content. For example, the relation "Document A is a basis for document B given argument C". From now on however, let us restrict ourselves to links in the conventional sense, that is, with two ends.

Should the links be monodirectional or bidirectional?

If they are bidirectional, a link always exists in the reverse direction. A disadvantage is that it might constrain the author of a hypertext. An advantage is that often, when a link is made between two nodes, it is made in one direction in the mind of its author, but another reader may be more interested in the reverse link. Bidirectional linking allows the system to deduce the inverse relationship, adding information for free.

KMS and HyperCard have one-way links; Enquire has two-way links. Automatic reverse links are useful for protected databases. See also: Private overlaid web, Generic Links.

A compromise is one-way links in the data model, with reverse links added automatically where allowed. See Building Back-links.

Should anchors have more than one link?

There is a design issue whether one anchor may lead to many links, and/or one link have many anchors. Many anchors leading to the same reference is reasonable. If one source anchor leads to multiple destinations, ambiguity can occur; menus could resolve this, but it complicates things. Many-to-one mapping is preferred.

JFG disagrees and prefers a small menu if ambiguous. Microcosm implements this.

Should links be typed?

A typed link carries semantic information for data management. A default "untyped" type normally exists. See also a list of some types. JFG suggests multiple types shouldn’t be represented in one link; TBL agrees. Link typing helps with graphical overviews and automatic tracing.

Should links contain ancillary information?

Links may include dates, versioning, authorship, or comments. This can complicate the data model, possibly making links into nodes themselves. Information could also be stored in source or intermediate nodes. Annotatable links may be considered but are not recommended.

Should a link contain Preview information?

Preview information helps readers decide whether to follow a link. This can be cached or included in the node itself, but it’s not essential. Fast graphic overviews can remove the need for preview functionality.
